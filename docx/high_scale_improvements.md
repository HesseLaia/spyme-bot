## SpyMe 高并发 / 高日活优化方向整理

目标：在现有代码基础上，逐步演进为可支撑 **几十万级日活 / 高并发** 的架构与实现。本文不要求一次性完成，而是作为中长期改造清单。

---

### 一、核心架构改造方向（必须项）

#### 1. 状态外置：从内存 Map 到外部存储

- **现状**：使用 `const games = new Map()` 存所有群的游戏状态，进程重启即全部丢失，且无法多实例共享。
- **目标**：抽象出统一的 `gameRepository`，将状态放到外部存储（推荐 Redis）。
- **建议实现步骤**：
  1. 定义接口（伪代码）：
     - `getGame(chatId): Promise<Game | null>`
     - `saveGame(game: Game): Promise<void>`
     - `deleteGame(chatId): Promise<void>`
  2. 在 `bot.js` 中，将所有对 `games` Map 的读写替换为 `gameRepository` 调用。
  3. 第一版可以先用内存实现（方便切换），确认无误后切换 Redis 实现。

#### 2. 多实例 & webhook 支持

- **现状**：单进程 Node + `bot.start()` 长轮询。
- **问题**：
  - 单实例成为吞吐和可靠性瓶颈。
- **目标**：支持多实例水平扩展，按需横向扩容。
- **建议实现步骤**：
  1. 切换为 Telegram webhook 模式（Grammy 支持）：
     - 在 `index.js` （或新建 `server.js`）中提供处理 update 的 HTTP 端点。
  2. 将服务挂在负载均衡后面（如 Nginx / Cloud LB），部署多个实例。
  3. 所有实例共享同一个 Redis / 数据库中的游戏状态。

#### 3. 并发安全与幂等性

- **现状**：单进程下基本安全，但在多实例 / 高并发环境下可能出现：
  - 同一局游戏被两个实例同时结算（`resolveVotes`）。
  - Telegram 重放 / 乱序 update 导致重复开局 / 重复结算。

- **目标**：保证对每个 `chatId` 的关键操作在全局是串行且幂等的。

- **建议实现**：
  - **分布式锁（按群加锁）**：
    - 使用 Redis 锁（例如基于 `SETNX` + 过期时间或 RedLock）：
      - 锁 key 示例：`spyme:lock:chat:${chatId}`。
      - 在 `doStart`、`resolveVotes`、`endGame` 等关键函数前后加锁 / 解锁。
  - **幂等状态机**：
    - 在 `Game` 中增加更明确的状态字段和版本号：
      - 如 `phase: "waiting" | "describing" | "voting" | "ended"`。
      - `version` 用于乐观锁（CAS）：读取时带出版本，保存时检查版本是否未变化。
    - 在 `resolveVotes` 等函数中，先检查：
      - 当前 `phase` 是否匹配；
      - 是否已经结算过本轮（如 `voteResolved === true`）。
    - 重复或延迟到达的 update 会被安全忽略。

### 二、性能与内存优化方向

#### 1. Game 结构瘦身与序列化优化

- **现状**：
  - `Game` 中包含较多字段（如 `players`、`votes`、若干 message id），对 Redis / DB 序列化时体积相对较大。
- **优化点**：
  - 区分「必须持久化」与「仅运行期」字段：
    - 如某些临时字段可以不写入 Redis。
  - 对 `votes` 等结构进行压缩：
    - 例如将 `{ [round]: { [voterId]: targetId } }` 的层级结构压缩为数组或更紧凑的结构。
  - 若后续需要进一步优化，可引入简单的压缩方案（如对文本较长字段压缩）。

#### 2. 控制无意义的高频输出

- **高频消息**：
  - 每一票目前都会 `ctx.reply` 一个确认消息，在高并发群数量下可能产生大量消息，增加 Telegram API 压力。
- **优化方式**：
  - 考虑只记录票数，不对每一票发群消息；
  - 或改为定时/阶段性汇总（例如“当前投票进度：X/Y 人已投”）。

- **日志输出**：
  - 高并发下大量 `console.log` 会影响性能和日志可用性。
  - 建议：
    - 使用结构化日志（JSON）；
    - 明确日志级别（info / warn / error），只在关键点输出。

### 三、容错与降级设计

#### 1. 外部依赖失败时的策略

- **Telegram API 调用失败**：
  - 如 `sendMessage` / `editMessageText` 超时或返回错误。
  - 建议：
    - 对关键消息（开局、结算）做有限次重试（如 1–3 次，带退避）。
    - 超过重试次数后：
      - 给群内发一条简短错误提示；
      - 尝试让本局进入一个“安全结束状态”（如强制 `ended` 并清理状态）。

- **Redis / 数据库出问题**：
  - 读失败时，避免做危险写操作；
  - 写失败时：
    - 可以短暂拒绝新开局；
    - 对进行中的局，记录告警日志，必要时标记该 `chatId` 不可继续，避免数据不一致。

#### 2. 单局异常自恢复

- 在 `resolveVotes`、`doStart`、`promptNextDescribe`、`startVotePhase` 等关键流程中，一旦捕获异常，应：
  - 尝试回滚或至少将 `Game.status` 标为 `ended`；
  - 从持久化中删除该局或标记为“异常结束”；
  - 在群中发出“本局出现错误，已终止，请重新 /spyme 开局”的提示。

---

### 四、限流与滥用防护

#### 1. 按用户 / 按群限流

- **目的**：防止恶意刷命令 / callback 拖垮服务或打满 Telegram API 的速率上限。
- **建议**：
  - 对关键操作设置简单速率限制，例如：
    - 单个用户在某群中，每秒可触发的 callback 次数上限；
    - 某个群在单位时间内的 `/spyme` 开局次数上限。
  - 可借助 Redis 的计数器（如 `INCR` + 过期时间）实现。

#### 2. 全局降级策略

- 当监控发现：
  - 平均响应时间大幅上升；
  - 错误率显著上升；
  - 或 Redis / 数据库压力过大时，
- 可以临时进入“降级模式”：
  - 拒绝新游戏创建，只允许已在进行中的局完成；
  - 或针对某些功能（例如额外文案、推广链接）暂时关闭，减小负载。

### 五、可观测性与运维

#### 1. 指标与监控

- 为关键链路埋点，输出包括但不限于：
  - 每秒处理的 update 数；
  - 各 handler 平均耗时（开局、投票、结算等）；
  - 外部依赖（Redis、Telegram API）的调用耗时与错误率；
  - 活跃游戏数量、每轮平均时长等。
- 结合 Prometheus + Grafana 或云厂商 APM 可视化这些指标。

#### 2. 日志体系

- 使用统一的日志函数替代直接 `console.log`：
  - 支持日志级别（info / warn / error）；
  - 输出结构化 JSON，便于收集和检索。
- 配合集中式日志系统（如 ELK / Loki）进行归集和检索。

### 六、代码结构与测试（为长期演进做准备）

#### 1. 进一步拆分 handlers

- **现状**：所有命令与 callback 仍集中在 `bot.js`。
- **目标**：提升可维护性和团队协作效率。
- **建议结构**：
  - `handlers/commands.js`：`/spyme`、`/join`、`/start`、`/end`、`/help`、`/ping`、`/status` 等。
  - `handlers/callbacks.js`：`join`、`start`、`desc_done_*`、`vote_*`、`new_game` 等 inline 按钮回调。
  - `index.js`：只负责加载配置、创建 `Bot`、注册各 handler、启动（或挂载 webhook）。

#### 2. 单元测试与集成测试

- **单元测试（优先针对纯逻辑部分）**：
  - 对 `game-state.js` 的函数（`createGame`、`assignRoles`、`checkWin` 等）编写单测：
    - 覆盖不同玩家数、卧底 / 白板组合、各种胜利条件。
- **集成测试**：
  - 模拟一整局流程（通过伪造 update 或直接调用 handler）：
    - `/spyme` → `/join` 多人 → `/start` → 描述 → 投票 → 结算。
  - 确保未来在重构与优化时不会破坏核心游戏体验。

---

### 七、实施建议与优先级

**近期（为高并发打基础）：**

1. 抽象 `gameRepository` 接口，先用内存实现，再切 Redis。
2. 在关键操作上增加幂等检查（利用现有的 `status` / `voteResolved` 字段）。
3. 为 `game-state` 写基础单元测试，锁定规则行为。

**中期（支撑多实例与更高负载）：**

1. 切 webhook + 负载均衡，实现多实例部署。
2. 在 `doStart`、`resolveVotes`、`endGame` 等关键处接入分布式锁。
3. 加入基本限流与错误重试机制。

**长期（运营与可观测性）：**

1. 接入监控与日志平台，完善告警。
2. 进一步拆分 handlers 结构，优化项目可读性与可维护性。
3. 根据业务发展情况再评估是否需要更多深度优化（如多区域部署、更多游戏模式等）。
