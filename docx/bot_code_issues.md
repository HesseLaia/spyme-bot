## SpyMe 机器人代码问题整理

本文件整理当前 `bot.js` 中存在的主要问题，方便后续重构与优化。

---

### 一、安全与配置

- **硬编码 BOT_TOKEN（严重安全问题）**
  - 在代码中直接写死真实的 Telegram Bot Token，一旦仓库泄露或被他人查看，就可以完全控制机器人。
  - 建议：
    - 使用环境变量（如 `process.env.BOT_TOKEN`）而不是硬编码。
    - 本仓库中出现过的真实 Token 需要在 BotFather 中立即重置，视为已经泄露。

- **其他配置硬编码**
  - `BOT_USERNAME`、`BLINK_BOT` 等直接写在代码里，不利于多环境部署和后期修改。
  - 建议：
    - 抽取到统一的配置文件或使用环境变量，如 `BOT_USERNAME`, `BLINK_BOT`.

---

### 二、代码结构与模块化

- **单文件过大**
  - 所有逻辑（常量、词库、游戏状态、命令处理、回调、文案、启动逻辑）都集中在一个约 500 行的 `bot.js` 中，耦合度高，可读性和可维护性差。
  - 建议：
    - 按职责拆分为多个模块，例如：
      - `word-pairs.js`：词库与 `randomPair`。
      - `game-state.js`：`createGame`、`assignRoles`、`getAlive`、`checkWin`、`endGame` 等纯游戏逻辑。
      - `handlers/commands.js`：所有命令（`/spyme`、`/join`、`/start`、`/end`、`/help`）。
      - `handlers/callbacks.js`：按钮回调（`join`、`start`、`desc_done_*`、`vote_*`、`new_game`）。
      - `index.js`：创建 `Bot`、加载配置、注册各 handler、调用 `bot.start()`。

- **文案与逻辑混杂**
  - 大段多语言文案直接写在 handler 函数内，增加阅读负担。
  - 建议：
    - 将文案抽离为单独的文案模块或模板函数，例如 `messages.buildWelcomeMessage`、`messages.buildEndGameMessage`。

- **缺少明确的分层结构**
  - 游戏规则与 Telegram 交互代码紧密混在一起，不利于单元测试。
  - 建议：
    - 将“纯业务逻辑”（与 Telegram 无关）和“适配层”（ctx / bot.api 调用）分层。

---

### 三、游戏流程与健壮性

- **玩家无法接收私聊词语时的处理不完整**
  - 在发放词语的 DM 阶段，如果某个玩家没有先私聊激活 bot 或屏蔽了 bot，会导致发送失败。
  - 当前逻辑只是群内提示“无法 DM 某玩家”，但仍然把该玩家保留在游戏中，继续描述和投票流程。
  - 风险：
    - 该玩家没有拿到词语，却仍然参与游戏，破坏游戏公平性。
  - 建议：
    - 统计 DM 成功与失败的玩家：
      - 对 DM 失败的玩家：从 `players` 中移除或标记为已经出局，并在群内明确提示。
      - 或者直接终止/重置本局，引导所有人先私聊 `/start` 再开局。

- **投票超时与提前投票完成之间的竞态**
  - `startVotePhase` 中使用 `setTimeout` 在 120 秒后调用 `resolveVotes`，同时当所有存活玩家都投票时也会立即调用 `resolveVotes`。
  - 虽然在 `resolveVotes` 开头有 `status` 检查，避免重复结算，但逻辑略显隐晦。
  - 建议：
    - 在 `game` 中保存投票定时器引用（如 `game.voteTimeout`），在第一次成功结算后 `clearTimeout` 并设置布尔标记（如 `game.voteResolved = true`），使流程更清晰可控。

- **游戏结束后的旧回调请求**
  - 游戏结束后使用 `games.delete(chatId)`，但用户可能迟点才点击旧按钮。
  - 目前大部分回调在 `!game` 时直接 return，不会报错，但用户体验不明确。
  - 建议：
    - 在相关 callback 中，当 `!game` 时给出统一反馈提示，例如“本局已结束，请发送 /spyme 开新局”。

### 四、游戏规则与业务设计

- **卧底人数和白板规则写死**
  - 当前规则：
    - 玩家数 ≥ 9 时有 2 个卧底，否则 1 个。
    - 玩家数 ≥ 7 时增加白板角色。
  - 问题：
    - 规则散落在 `assignRoles` 内部，修改不直观。
  - 建议：
    - 抽取为独立的规则函数，例如：
      - `getSpyCount(playerCount)`
      - `hasBlankRole(playerCount)`
    - 方便后续调整和配置化。

- **白板胜利条件简单，缺少“猜词”步骤**
  - 当前胜利判定为：若场上仍有白板、无卧底且存活人数 ≤ 2，则判定白板获胜。
  - 若设计期望是“白板需要猜中词语才能获胜”，现有流程中缺少显式的“白板猜词”阶段。
  - 建议：
    - 根据产品设计补充白板猜词流程，或在文档中明确当前简化版规则。

- **描述阶段完全依赖玩家自觉**
  - 当前机制：玩家只需点击“描述完成”按钮即可进入下一位描述，不校验是否实际发言。
  - 虽然这样实现简单、易玩，但也可能导致恶意乱点。
  - 建议（可选）：
    - 若希望更严谨，可在回调中检查最近若干条群消息中是否有该玩家发言（实现复杂度较高，可暂不做）。

### 五、可读性与风格问题

- **部分命名不够语义化**
  - 例如：`doStart` 命名较含糊，可以改为 `startGameInGroup` 等更有语义的函数名。
  - `WORD_PAIRS` 中字段使用 `c`、`s`（civilian / spy），对于阅读者不够直观。
  - 建议：
    - 使用更明确的命名，例如 `civilianWord`、`spyWord`。

- **缺少类型约束**
  - 游戏状态对象 `game` 结构复杂（包含 players、votes、round 等），纯 JavaScript 容易在维护时属性名写错。
  - 建议：
    - 至少为关键数据结构添加 JSDoc 类型注释。
    - 长期来看可以迁移到 TypeScript，为 `ctx`、`game` 等提供完整类型检查。

- **文件内部结构可以更有层次**
  - 目前命令注册、回调注册、游戏逻辑函数分布比较随意。
  - 建议：
    - 固定顺序：常量与工具函数 → 游戏纯逻辑 → 命令 handler → 回调 handler → 启动逻辑。

---

### 六、运行与部署层面的限制

- **游戏状态只存在于内存 Map 中**
  - 机器人重启后，所有正在进行的游戏都会丢失。
  - 后续若做多实例部署（多进程 / 多节点），各实例之间的游戏状态无法共享。
  - 建议：
    - 视需求考虑引入轻量的状态存储（如 Redis），或至少在代码层抽象出 `gameRepository` 接口，方便未来替换实现。

- **使用长轮询（long polling）启动方式**
  - `bot.start()` 默认使用长轮询，适合简单部署，但不适合所有生产环境。
  - 建议（长期）：
    - 如需部署到 serverless 或需要更好控制的环境，可考虑改为 webhook 方式。

---

### 七、其他补充建议

- **统一的帮助与状态检查命令**
  - 目前有 `/help`，但缺少简易健康检查命令，如 `/ping`、`/status`。
  - 建议：
    - 增加简单命令用于：
      - 检查机器人是否在线。
      - 输出版本号或当前配置模式（测试/生产）。

- **推广文案与游戏功能解耦**
  - 代码中嵌入了 Blink 推广链接和文案，直接写在游戏结束消息中。
  - 建议：
    - 将推广相关内容抽象成可配置项或独立模块，以便调整投放策略或多渠道区分。

---

以上问题和建议可作为后续重构与优化的参考清单，优先级上建议先解决安全与配置问题（BOT_TOKEN 和配置抽离），再逐步进行模块化和流程健壮性改造。
